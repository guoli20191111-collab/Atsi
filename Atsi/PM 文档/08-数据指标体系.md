## 一、北极星指标(One Metric That Matters)

### 1.1 选择

> **北极星指标:分享率 = 完测用户中点击分享卡片下载/分享的比例**

### 1.2 选择理由

ATSI 的核心价值假设是:**"用户愿意把测试结果分享给朋友"**——这代表内容真的戳中了用户、品牌已经成为社交货币。其他指标(完测率、复测率)虽然重要,但都是"过程指标",分享率才是"价值兑现指标"。

### 1.3 目标值

| 时间 | 分享率目标 |
|---|---|
| V2 上线后 4 周 | ≥ 15%(基线建立) |
| V2 上线后 12 周 | ≥ 25%(对标头部测试产品) |
| V2 上线后 24 周 | ≥ 30%(成为爆款的门槛) |

### 1.4 监控频率

- 每日观察(早晨开始查前一日数据)
- 每周复盘(周日晚上看周环比)
- 每月归因(月底分析"分享率变化的主要原因")

## 三、关键指标卡(Dashboard 设计)

### 3.1 一级看板(每日)

| 指标 | 计算口径 | 目标 |
|---|---|---|
| **PV** | 落地页 page_view_landing 总数 | 周环比 +20% |
| **UV** | 去重独立用户数 | 周环比 +15% |
| **进入测试率** | click_start_test / page_view_landing | ≥ 50% |
| **完测率** | quiz_complete / click_start_test | ≥ 60% |
| **分享率(北极星)** | result_share_click / quiz_complete | ≥ 25% |
| **下载卡片率** | result_share_card_download / quiz_complete | ≥ 15% |
| **重测率(同日)** | result_retry / quiz_complete(当日)| ≤ 5%(过高可能是匹配度差) |

### 3.2 二级看板(每周)

| 指标 | 计算口径 | 目标 |
|---|---|---|
| **7 日复测率** | 7 日内重测的 UV / 7 日前完测 UV | ≥ 15% |
| **完测平均用时** | quiz_complete 事件中的总用时均值 | 4-6 分钟 |
| **人格分布均匀度** | 30 个人格被匹配的频次方差 | 越低越好,设阈值 |
| **风格分布(用户答案)** | A/B/C/D 风格在用户答案中的占比 | 25% ± 5% |
| **维度差异度** | 用户 15 维度向量的分散度 | 越高越好(避免大部分用户测出同一类) |
| **来源渠道分布** | 各 referrer 占比 | 小红书 > 微信 > 直接访问 |

### 3.3 三级看板(每月)

| 指标 | 计算口径 | 目标 |
|---|---|---|
| **30 日留存率** | 30 日内回访 UV / 30 日前注册 UV | ≥ 8% |
| **匹配度感知均分** | 用户主观评分(1-5)均值 | ≥ 4.0 |
| **NPS** | 净推荐值 = 推荐者% - 贬损者% | ≥ 30 |
| **30 人格收藏度** | 各人格被收藏到 localStorage 的频次 | 找出"明星人格" |
| **人格百科 PV** | persona_wiki_view 各人格 PV | 找出"出圈人格" |

## 四、埋点设计(完整表)

### 4.1 页面埋点

| 事件名 | 触发 | 字段 |
|---|---|---|
| `page_view_landing` | 落地页加载完成 | `referrer` `device` `user_agent` `session_id` |
| `page_view_quiz` | 答题页加载 | `session_id` |
| `page_view_result` | 结果页加载 | `session_id` `persona_code` |
| `page_view_wiki` | 人格百科页加载 | `persona_code` `entry_source` |

### 4.2 行为埋点

| 事件名 | 触发 | 字段 |
|---|---|---|
| `click_start_test` | 点击"开始测试" | `landing_stay_time_sec` |
| `quiz_question_view` | 每题加载 | `question_id` `question_dim` `question_index` |
| `quiz_question_answer` | 每题作答 | `question_id` `option` `answer_time_sec` |
| `quiz_back` | 返回上一题 | `question_id` |
| `quiz_quit` | 中途退出(beforeunload) | `last_question_index` |
| `quiz_complete` | 完成 30 题 | `total_time_sec` `style_vector` `answer_log` |
| `result_share_click` | 点击分享按钮 | `share_channel`(wechat/xhs/dy/weibo/copy) |
| `result_share_card_download` | 下载分享卡片 | `card_size`(square/portrait/landscape) |
| `result_retry` | 点击重新测试 | `last_persona_code` |
| `result_rate` | 用户给匹配度打分 | `rating`(1-5) |
| `result_jump_wiki` | 从结果页跳转人格百科 | `from_persona` `to_persona` |

### 4.3 用户属性(localStorage 维度)

| 属性 | 用途 |
|---|---|
| `user_id`(本地生成) | 跨 session 跟踪 |
| `first_test_date` | 首次测试日期 |
| `test_count` | 累计测试次数 |
| `persona_history` | 历史匹配人格列表 |
| `last_share_date` | 最近分享日期 |

## 五、数据回收方案

### 5.1 技术方案选择

| 方案 | 成本 | 优点 | 缺点 |
|---|---|---|---|
| **A. 自建简单埋点(推荐 V2)** | < 100 元/月 | 完全可控,数据归自己 | 需要写后端接收 + 简单看板 |
| **B. Google Analytics** | 免费 | 上手快 | 中国访问受限,数据归 Google |
| **C. 神策 / Mixpanel(付费)** | 100-1000 元/月 | 看板成熟 | 成本高,小流量阶段不划算 |
| **D. Plausible / Umami(轻量分析)** | 免费/低成本 | 隐私友好 | 自定义事件能力一般 |

**建议 V2 阶段走 A 方案**:用一个简单的 Node.js + SQLite + Vue 看板,自己写 200-300 行代码即可。

### 5.2 数据保存与隐私

- 不存储任何个人身份信息(无登录)
- 用户 ID = 本地生成的 UUID,可被用户清空
- 答题数据匿名化,用于内容优化
- 隐私声明在落地页底部链接

## 六、A/B 测试设计(V2 内置)

### 6.1 关键 A/B 测试假设

| # | 假设 | 测试设计 | 关键指标 |
|---|---|---|---|
| AB1 | "新标语 vs 原标语 哪个完测率高" | 落地页 50/50 分流 | 进入测试率 |
| AB2 | "分享卡片 V1 设计 vs V2 设计" | 50/50 分流 | 分享率 |
| AB3 | "解读 250 字 vs 350 字 哪个分享率高" | 50/50 分流 | 分享率 |
| AB4 | "结果页评分组件 显示 vs 不显示" | 50/50 分流 | 复测率、匹配度感知 |

### 6.2 A/B 测试执行机制

- 用户进入时根据 user_id 哈希分组
- 一个用户在同一实验中保持同一组别
- 实验最少跑 2 周或样本量 ≥ 500/组

## ⚠️ 自用反思

1. **完测率 60% / 分享率 25% 等目标值都是经验估**,可能过乐观,V2 第一周后需要根据实际基线重新校准
2. **"人格分布均匀度阈值"没有具体数字**,需要根据 V2 第一周数据来定参考线
3. **A/B 测试需要足够样本量**,V2 前 4 周流量可能不够支撑多个并行实验,要排优先级
4. **"匹配度感知评分"需要用户主动参与**,但用户答完结果页不一定会去打分——评分组件设计上要"友好但不强制"
5. **OKR 里的目标值缺乏数据基线参考**,小流量阶段的 OKR 容易脱节,建议 V2 上线 2 周后做一次校准
6. **没有规划"数据驱动内容迭代"的具体流程**——比如某个人格被匹配率特别低,应该怎么处理?需要补一个 SOP
